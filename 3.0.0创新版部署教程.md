# 创新版 3.0.0 部署教程

> 本教程适用于直接在 Cloudflare Worker 在线编辑器中粘贴代码部署，无需安装 Wrangler，也无需使用命令行上传代码。

---

## 一、部署前准备

部署前请准备以下内容：

| 配置名称 | 作用 |
| --- | --- |
| `BOT_TOKEN` | Telegram 机器人 Token |
| `WEBHOOK_SECRET` | Telegram Webhook 请求验证密钥 |
| `ADMIN_IDS` | 主管理员 Telegram 数字 ID |
| `ADMIN_GROUP_ID` | 接收用户消息的管理群 ID |
| `TG_BOT_DB` | Cloudflare D1 数据库绑定名称 |

管理群需要满足以下条件：

- 管理群必须是 Telegram 超级群；
- 管理群必须开启“话题”功能；
- 机器人必须已经加入管理群；
- 机器人必须被设置为管理员；
- 机器人需要拥有“发送消息”权限；
- 机器人需要拥有“管理话题”权限；
- 机器人需要拥有“置顶消息”权限。

---

## 二、创建 D1 数据库

登录 Cloudflare 控制台，依次进入：

```text
存储和数据库
→ D1 SQL 数据库
→ 创建数据库
```

数据库名称可以填写：

```text
tg-bot-db
```

数据库创建完成后，不需要手动执行建表 SQL。

创新版 3.0.0 第一次运行时会自动创建以下数据表：

```text
config
users
messages
processed_updates
```

同时会自动创建必要的数据库索引，并尝试兼容旧版本数据库中缺少的字段。

---

## 三、创建 Cloudflare Worker

进入 Cloudflare 控制台：

```text
Workers 和 Pages
→ 创建
→ 创建 Worker
```

Worker 名称可以自定义，例如：

```text
tgbot
```

创建完成后，点击：

```text
编辑代码
```

删除在线编辑器中的默认示例代码，然后粘贴创新版 3.0.0 的完整代码。

确认代码入口格式为：

```const USER_STATE = {
  NEW: 'new',
  PENDING: 'pending_verification',
  VERIFIED: 'verified'
};
```

粘贴完成后，点击右上角的：

```text
部署
```

> 首次部署时，由于环境变量和 D1 数据库绑定尚未配置，访问 Worker 可能会显示初始化错误，这是正常现象。

---

## 四、添加 Worker 环境变量

进入对应 Worker 的：

```text
设置
→ 变量和机密
```

添加以下四个变量。

### 1. 添加 `BOT_TOKEN`

变量名称：

```text
BOT_TOKEN
```

变量值：

```text
你的 Telegram 机器人 Token
```

建议将该变量类型设置为：

```text
机密
```

---

### 2. 添加 `WEBHOOK_SECRET`

变量名称：

```text
WEBHOOK_SECRET
```

变量值需要填写一个随机且足够长的安全字符串。

可以在 UUID网站中生成一个UUID串 ：

```https://1024tools.com/uuid ""
```

复制命令输出的随机字符串，并将其保存到：

```text
WEBHOOK_SECRET
```

建议将该变量类型设置为：

```text
机密
```

> 后续设置 Telegram Webhook 时，`secret_token` 必须与这里填写的 `WEBHOOK_SECRET` 完全一致。

---

### 3. 添加 `ADMIN_IDS`

变量名称：

```text
ADMIN_IDS
```

变量值填写主管理员的 Telegram 数字 ID。

单个主管理员示例：

```text
123456789
```

多个主管理员使用英文逗号分隔：

```text
123456789,987654321
```

注意事项：

- 不要填写 Telegram 用户名；
- 不要添加引号；
- 使用英文逗号分隔；
- 不要在 ID 中添加其他字符。

---

### 4. 添加 `ADMIN_GROUP_ID`

变量名称：

```text
ADMIN_GROUP_ID
```

变量值填写接收用户消息的管理群 ID，例如：

```text
-1001234567890
```

注意事项：

- 不要添加引号；
- 管理群必须开启话题功能；
- 机器人必须拥有管理话题权限。

---

## 五、绑定 D1 数据库

进入对应 Worker 的：

```text
设置
→ 绑定
→ 添加绑定
→ D1 数据库
```

填写以下内容：

```text
变量名称：TG_BOT_DB
D1 数据库：选择之前创建的 tg-bot-db
```

绑定变量名称必须严格填写为：

```text
TG_BOT_DB
```

请注意区分大小写，不能修改为：

```text
DB
TG_DB
tgbot-db
```

代码只会读取：

```javascript
env.TG_BOT_DB
```

因此绑定名称不正确时，机器人无法访问数据库。

添加完成后，保存绑定。

### D1 绑定失败的处理方法

如果出现以下错误：

```text
D1 binding references database which was not found
```

请检查：

1. Worker 和 D1 数据库是否位于同一个 Cloudflare 账户；
2. 所选择的 D1 数据库是否仍然存在；
3. 刷新 Cloudflare 控制台后重新选择数据库；
4. 删除已经失效的旧数据库；
5. 创建一个新的 D1 数据库；
6. 使用 `TG_BOT_DB` 作为变量名称重新添加绑定。

如果旧数据库条目存在但无法绑定，可以创建一个新数据库，例如：

```text
tg-bot-db-new
```

然后刷新 Worker 页面并重新绑定。

---

## 六、重新部署 Worker

配置完环境变量和 D1 数据库绑定后，重新进入：

```text
Worker
→ 编辑代码
```

点击右上角的：

```text
部署
```

确保最新代码、环境变量和数据库绑定全部生效。

修改以下内容后，建议重新部署 Worker：

- Worker 代码；
- `BOT_TOKEN`；
- `WEBHOOK_SECRET`；
- `ADMIN_IDS`；
- `ADMIN_GROUP_ID`；
- D1 数据库绑定。

---

## 七、测试 Worker 是否正常运行

复制 Worker 的公开访问地址，例如：

```text
https://你的Worker地址/
```

使用浏览器访问该地址。

也可以在 PowerShell 中执行：

```powershell
Invoke-WebRequest `
  -Uri "https://你的Worker地址/" `
  -Method Get
```

正常情况下应返回：

```text
Telegram Bot Worker is running.
```

出现该内容通常说明：

- Worker 代码已经成功部署；
- 必要环境变量已经配置；
- D1 数据库已经正确绑定；
- 数据库初始化和迁移已经完成。

### 初始化错误

如果返回：

```text
Initialization Error
```

请根据后面的错误信息检查配置。

例如：

```text
缺少环境变量或绑定：TG_BOT_DB
```

说明 D1 数据库没有正确绑定，或者绑定变量名称填写错误。

如果提示缺少其他变量，请检查：

```text
BOT_TOKEN
WEBHOOK_SECRET
ADMIN_IDS
ADMIN_GROUP_ID
```

## 八、设置 Telegram Webhook

确认 Worker 地址可以正常访问后，打开 Windows PowerShell，执行以下命令：

```powershell
$botToken = "你的BOT_TOKEN"
$webhookSecret = "与Worker中WEBHOOK_SECRET完全一致的值"
$webhookUrl = "https://你的Worker地址/"

$body = @{
    url = $webhookUrl
    secret_token = $webhookSecret
    allowed_updates = @(
        "message"
        "edited_message"
        "callback_query"
    )
    drop_pending_updates = $true
} | ConvertTo-Json -Depth 5

Invoke-RestMethod `
    -Uri "https://api.telegram.org/bot$botToken/setWebhook" `
    -Method Post `
    -ContentType "application/json" `
    -Body $body
```

设置成功后会返回类似内容：

```text
ok   result description
--   ------ -----------
True True   Webhook was set
```

这表示 Telegram 已经成功保存 Webhook。

### 关于 `drop_pending_updates`

首次部署或从旧版本升级时，建议使用：

```powershell
drop_pending_updates = $true
```

这样会清除部署失败期间积压的旧消息，避免部署完成后集中处理旧消息。

正常运行后，如果只是修改 Webhook 地址，可以改成：

```powershell
drop_pending_updates = $false
```

### Webhook Secret 注意事项

以下两个值必须完全一致：

```text
Cloudflare Worker 环境变量中的 WEBHOOK_SECRET
```

和：

```text
Telegram setWebhook 请求中的 secret_token
```

如果不一致，Worker 会拒绝 Telegram 请求，并返回：

```text
401 Unauthorized
```

## 九、检查 Webhook 状态

设置完成后，在 PowerShell 中执行：

```powershell
$info = Invoke-RestMethod `
  -Uri "https://api.telegram.org/bot$botToken/getWebhookInfo"

$info.result | Format-List *
```

正常情况下应看到：

```text
url                  : https://你的Worker地址/
pending_update_count : 0
allowed_updates      : {message, edited_message, callback_query}
```

重点检查以下字段：

```text
last_error_message
```

正常情况下，该字段应该为空。

### 出现 `401 Unauthorized`

如果显示：

```text
Wrong response from the webhook: 401 Unauthorized
```

说明 Webhook Secret 不一致。

请确认以下两处使用的是完全相同的值：

1. Worker 环境变量 `WEBHOOK_SECRET`；
2. `setWebhook` 中的 `secret_token`。

修改后需要：

1. 保存 Worker 环境变量；
2. 重新部署 Worker；
3. 重新执行 `setWebhook`。

### 出现 `500 Internal Server Error`

如果显示：

```text
Wrong response from the webhook: 500 Internal Server Error
```

请检查以下配置是否完整：

```text
BOT_TOKEN
WEBHOOK_SECRET
ADMIN_IDS
ADMIN_GROUP_ID
TG_BOT_DB
```

也可以直接访问 Worker 地址，查看具体的初始化错误提示。

### `pending_update_count` 持续增加

如果：

```text
pending_update_count
```

持续增加，通常表示 Telegram 无法正常将消息发送到 Worker。

请检查：

- Worker 地址是否填写正确；
- Worker 地址是否可以公开访问；
- Webhook URL 是否存在拼写错误；
- 自定义域名是否可以正常解析；
- Worker 是否返回 `401`；
- Worker 是否返回 `500`；
- `WEBHOOK_SECRET` 是否一致。

## 十、首次运行测试

部署完成后，建议按照以下顺序测试。

### 1. 测试 `/start`

在机器人私聊中发送：

```text
/start
```

普通用户应该依次收到：

1. 欢迎消息；
2. 人机验证问题。

主管理员发送 `/start` 时，应该显示机器人配置菜单。

---

### 2. 测试人机验证

发送正确的验证答案。

机器人应该回复：

```text
🎉 验证成功！现在可以开始发送消息了。
```

---

### 3. 测试用户话题

验证完成后，向机器人发送一条普通消息。

机器人应该执行以下操作：

1. 在管理群中创建一个新的用户话题；
2. 在话题中发送用户资料卡；
3. 将用户消息复制到对应话题；
4. 将用户 ID 与话题 ID 保存到 D1 数据库。

如果没有创建话题，请检查：

- `ADMIN_GROUP_ID` 是否正确；
- 管理群是否开启话题功能；
- 机器人是否为群管理员；
- 机器人是否拥有管理话题权限；
- 机器人是否拥有发送消息权限。

---

### 4. 测试管理员回复

在管理群对应的用户话题中发送一条消息。

机器人应该将该消息复制到用户私聊。

支持转发的内容包括：

- 文本；
- 图片；
- 视频；
- 文件；
- 语音；
- 音频；
- 贴纸；
- GIF；
- 带格式的文本；
- 带 Caption 的媒体消息。

---

### 5. 测试管理菜单

主管理员在机器人私聊中发送：

```text
/start
```

机器人应该显示配置管理菜单。

管理菜单包含：

- 基础配置；
- 自动回复管理；
- 关键词屏蔽管理；
- 消息类型过滤；
- 协管员授权设置。

---

```text

3. 在在线编辑器中粘贴创新版3.0.0完整代码

5. 添加BOT_TOKEN

7. 添加ADMIN_IDS

10. 重新部署Worker
11. 访问Worker地址并确认运行正常
12. 使用PowerShell设置Telegram Webhook
13. 使用getWebhookInfo检查Webhook状态

```

## 十二、常见问题

### 1. 发送 `/start` 没有反应

请依次检查：

```text
Worker地址是否可以正常访问
BOT_TOKEN是否正确
WEBHOOK_SECRET是否一致
TG_BOT_DB是否已经绑定
Webhook地址是否正确
getWebhookInfo是否存在错误
```

重新部署后，需要向机器人发送一条新的 `/start`。

之前已经发送的消息不会因为重新部署而自动重新处理。

---

### 2. Worker 显示初始化错误

检查是否已经配置：

```text
BOT_TOKEN
WEBHOOK_SECRET
ADMIN_IDS
ADMIN_GROUP_ID
TG_BOT_DB
```

其中：

```text
TG_BOT_DB
```

不是普通环境变量，而是 Cloudflare D1 数据库绑定。

---

### 3. 验证成功但没有创建用户话题

请检查：

- `ADMIN_GROUP_ID` 是否正确；
- 管理群是否为超级群；
- 管理群是否已经开启话题功能；
- 机器人是否已经加入管理群；
- 机器人是否已经设置为管理员；
- 机器人是否拥有管理话题权限；
- 机器人是否拥有发送消息权限。

---

### 4. 管理员回复无法发送给用户

请检查：

- 回复是否发送在用户对应的话题中；
- 管理员 ID 是否存在于 `ADMIN_IDS`；
- 是否已经通过管理菜单添加为协管员；
- 机器人是否仍然可以向用户发送消息；
- 用户是否已经停止或屏蔽机器人。

---

### 5. D1 数据库绑定失败

请检查：

- D1 数据库和 Worker 是否位于同一个 Cloudflare 账户；
- D1 数据库是否真实存在；
- 绑定变量名称是否严格为 `TG_BOT_DB`；
- Cloudflare 页面是否仍然保留了失效的数据库缓存；
- 删除失效数据库后，重新创建并绑定一个新数据库。

---

### 6. Webhook 返回 `401`

通常是以下原因：

```text
WEBHOOK_SECRET不一致
```

请将同一个 Secret 同时填写到：

1. Worker 的 `WEBHOOK_SECRET`；
2. `setWebhook` 的 `secret_token`。

保存后重新部署 Worker，并重新设置 Webhook。

---

### 7. Webhook 返回 `500`

通常是以下原因：

- 环境变量缺失；
- D1 数据库未绑定；
- D1 绑定名称错误；
- 数据库初始化失败；
- Worker 中的 Bot Token 无效；
- Worker 代码部署不完整。

建议先使用浏览器访问 Worker 地址，根据返回的错误内容进行检查。

---

### 8. 管理菜单无法打开

请检查当前账号的 Telegram 数字 ID 是否已经填写到：

```text
ADMIN_IDS
```

多个主管理员必须使用英文逗号分隔：

```text
123456789,987654321
```

不要填写 Telegram 用户名。

---

### 9. 用户消息没有被转发

请检查：

- 用户是否已经完成验证；
- 用户是否已经被屏蔽；
- 当前消息类型是否被过滤；
- 消息是否命中了屏蔽关键词；
- 管理群 ID 是否正确；
- 机器人是否拥有管理群发送消息权限；
- 机器人是否拥有创建话题权限。

---

## 十三、从旧版本升级

如果从旧版本升级到创新版 3.0.0，请按照以下步骤操作：

1. 备份旧版本 Worker 代码；
2. 保留原有 D1 数据库；
3. 进入 Worker 在线编辑器；
4. 删除旧版本代码；
5. 粘贴创新版 3.0.0 完整代码；
6. 确认已添加 `WEBHOOK_SECRET`；
7. 确认 D1 数据库绑定名称为 `TG_BOT_DB`；
8. 点击“部署”；
9. 访问 Worker 地址检查运行状态；
10. 使用相同的 `WEBHOOK_SECRET` 重新设置 Webhook；
11. 首次重设时使用 `drop_pending_updates = $true`；
12. 向机器人发送新的 `/start` 进行测试。

创新版 3.0.0 会自动检查并补充旧数据库缺少的字段，一般不需要手动删除或重建数据库。

> 为了防止数据丢失，升级前不建议删除原有 D1 数据库。

---

## 十四、安全注意事项

请勿公开以下内容：

```text
BOT_TOKEN
WEBHOOK_SECRET
Cloudflare API Token
数据库相关凭据
```

不要将真实 Token 或 Secret：

- 写入 GitHub 公开仓库；
- 写入 README；
- 放进公开截图；
- 发送到公开群聊；
- 直接写死在 Worker 代码中。

如果 `BOT_TOKEN` 曾经泄露，请通过 `@BotFather` 撤销并重新生成。

如果 `WEBHOOK_SECRET` 曾经泄露，请重新生成一个随机 Secret，并同时更新：

```text
Worker环境变量中的WEBHOOK_SECRET
Telegram Webhook中的secret_token
```

更新完成后，重新部署 Worker，并重新设置 Webhook。

## 十五、部署完成标准

满足以下条件，即可认为创新版 3.0.0 部署成功：

- [ ] Worker 地址可以正常访问；
- [ ] Worker 返回 `Telegram Bot Worker is running.`；
- [ ] 已添加 `BOT_TOKEN`；
- [ ] 已添加 `WEBHOOK_SECRET`；
- [ ] 已添加 `ADMIN_IDS`；
- [ ] 已添加 `ADMIN_GROUP_ID`；
- [ ] D1 数据库绑定名称为 `TG_BOT_DB`；
- [ ] `setWebhook` 返回 `Webhook was set`；
- [ ] `getWebhookInfo` 没有错误信息；
- [ ] `/start` 可以正常返回欢迎消息和验证问题；
- [ ] 用户可以正常完成人机验证；
- [ ] 用户消息可以进入管理群独立话题；
- [ ] 管理员可以在话题中回复用户；
- [ ] 主管理员可以正常打开配置菜单；
- [ ] 自动回复、关键词过滤和消息类型过滤可以正常使用。
